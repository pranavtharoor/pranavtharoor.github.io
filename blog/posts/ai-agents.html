The Strange Reality of Building AI Agents

Every few months, a new demo makes the rounds. An AI agent browses the web, books flights, files taxes, writes and deploys code — all in a single three-minute video. The comments light up. "AGI is here." "Engineers are finished." "Why would anyone hire a developer in five years?"

I've been building AI agents in production for over two years now. The reality is simultaneously more exciting and more humbling than any demo suggests.

What a demo gets right

The demos aren't lying. The raw capability is genuinely astonishing. Large language models can reason across long contexts, generate syntactically correct code in a dozen languages, infer intent from ambiguous instructions, and chain together multi-step plans with a coherence that would have seemed like science fiction five years ago.

When you wire a capable model to the right tools — a code interpreter, a search API, the ability to write files and call services — the emergent behavior is striking. Watching an agent decompose a vague instruction into concrete sub-tasks, execute them in sequence, and self-correct when something breaks is genuinely new territory for software.

The gap isn't in what AI agents can do. It's in how reliably they do it when the world doesn't cooperate.

What a demo leaves out

// reliability

Production software isn't a controlled demo. Users don't always provide clear inputs. External APIs return unexpected responses. Systems have partial failures. In a traditional software stack, you write code that handles these cases explicitly — validation, retries, fallbacks. An agent must reason its way through them.

Sometimes it does so brilliantly, recovering from an edge case you never anticipated. Other times it confidently makes exactly the wrong call. The non-determinism that makes LLMs feel "intelligent" is the same property that makes them hard to trust at the tail end of a distribution. And in production, the tail is where things break.

// evaluation

Traditional software is easy to test — you define inputs and expected outputs, you write assertions, you run the suite. Agents don't have right answers; they have better and worse answers along dimensions that are hard to quantify: correctness, efficiency, tone, the degree to which they stayed within the intended scope.

Evaluation becomes one of the hardest parts of the job. You end up building parallel infrastructure just to answer the question: is this version better than the last one? Human raters, LLM-as-judge pipelines, golden datasets curated by hand — it's real engineering, unglamorous and essential.

// prompt brittleness

Everyone who has worked with LLMs at depth has a story about a prompt that worked perfectly in development and fell apart in production. An extra sentence in the system prompt. A slightly different phrasing of the user request. A change in model version that altered the token distribution in ways you couldn't predict.

Prompt engineering gets dismissed as "not real engineering." That's wrong. Figuring out how to consistently elicit a specific behavior from a probabilistic system — across diverse inputs, across model updates, across the long tail of user intent — is a genuinely hard problem. It just doesn't look like the engineering we're used to.

What this actually changes

None of this is an argument that AI agents aren't transformative. They are. The question is what the transformation looks like from inside the engineering work.

The most useful mental model I've found: an AI agent is not a replacement for deterministic software. It's a new primitive that handles the parts of a system that were previously too ambiguous or too varied to automate at all. The skill is in knowing where to trust it and where to keep a human — or a traditional algorithm — in the loop.

That judgment doesn't come from watching demos. It comes from shipping things, watching them fail in interesting ways, and building up an intuition for the shape of the problem space. Which, if you think about it, is exactly how we learned to build reliable software in the first place.

The most honest thing I can say about working in this space is that I find it genuinely exciting — not because AI is magic, but because it isn't. The hard problems are still hard. The craft is still real. The bar for what "working" means is just higher, and more interesting, than it used to be.
